# MIPS Pseudoinstructions and Calling Convention

This note covers two parts of introductory MIPS that are easy to confuse at first:
- pseudoinstructions,
- and register roles under the calling convention.

These are related but not identical.

Pseudoinstructions explain:
- what assembly programmers are allowed to write,
- even when the hardware does not directly implement that exact mnemonic.

The calling convention explains:
- how registers are supposed to be used across function calls,
- so separately written procedures can interoperate safely.

## Pseudoinstructions

A pseudoinstruction looks like a MIPS instruction in assembly source, but it is not necessarily a real hardware instruction in the core ISA.

Instead, the assembler expands it into one or more real instructions.

So there is an important distinction:

```text
assembly syntax accepted by the assembler
!=
the exact instruction set directly implemented in hardware
```

Pseudoinstructions exist to make assembly programming easier and more readable.

## Why Pseudoinstructions Exist

The core ISA is intentionally small and regular.
That is good for hardware, but it means some useful high-level assembly actions are awkward to spell directly.

So the assembler provides shorthand forms for common patterns.

This lets the programmer write:
- clearer intent,
- fewer low-level steps,
- and less repetitive boilerplate,

while still targeting the same hardware instruction set.

## `blt`, `bgt`, `ble`, `bge`

These are classic pseudoinstructions for relational branching.

Examples:

```text
blt $s1, $s2, Label
bgt $s1, $s2, Label
ble $s1, $s2, Label
bge $s1, $s2, Label
```

The important point is that introductory MIPS does not have all of these as direct primitive branch instructions in the small core set.

Instead, the assembler can expand them using comparison plus branch.

### Example: `blt`

Conceptual meaning:

```text
if ($s1 < $s2) go to Label
```

Possible expansion:

```text
slt $at, $s1, $s2
bne $at, $zero, Label
```

Causal interpretation:
1. compare `$s1` and `$s2`
2. place the Boolean-style result in `$at`
3. branch if that result is nonzero

So `blt` is not magic.
It is shorthand for a real instruction sequence.

The same logic applies to:
- `bgt`
- `ble`
- `bge`

Each one is just a convenient assembler-level surface over a small number of true ISA instructions.

## `li`: Load Immediate

`li` means:

```text
load an immediate constant into a register
```

Example:

```text
li $t0, 42
```

If the constant is small enough, the assembler may expand it into something like:

```text
addi $t0, $zero, 42
```

If the constant needs more than the available immediate range, the assembler may construct it with multiple instructions such as:

```text
lui $t0, upper16
ori $t0, $t0, lower16
```

So `li` does not mean:

```text
the hardware has one universal constant-loading instruction for every possible value
```

It means:

```text
the assembler will generate whatever real instruction sequence is needed to place this constant in the target register
```

## `move`

`move` is another pseudoinstruction.

Example:

```text
move $t0, $s1
```

Typical expansion:

```text
add $t0, $s1, $zero
```

Meaning:

```text
$t0 = $s1 + 0
```

So `move` is not a fundamentally different machine operation.
It is a cleaner way to say:

```text
copy this register value
```

## What Pseudoinstructions Teach

Pseudoinstructions teach an important systems lesson:

- the assembler interface is slightly richer than the bare hardware ISA,
- and human-friendly assembly can still compile to a smaller regular machine vocabulary.

So when reading assembly, you must always know whether a mnemonic is:
- a true ISA instruction,
- or an assembler convenience that expands into multiple instructions.

## Register Names and Their Roles

MIPS registers are not only numbered.
They are also assigned conventional roles.

The calling convention gives each register family a purpose so code generated by one compiler or one function can safely interact with another.

This is not merely style.
It is a contract about:
- where arguments go,
- where results come back,
- which registers may be clobbered,
- and which registers must be preserved.

## `$zero`

Register `0`, written as `$zero`, always contains:

```text
0
```

It cannot be overwritten.

This is useful for:
- comparisons against zero,
- adding zero to copy values,
- building constants,
- and branch logic.

## `$at`

Register `1`, written as `$at`, is the **assembler temporary**.

It is commonly used by the assembler when expanding pseudoinstructions.

This is why hand-written assembly must treat `$at` carefully.
If the assembler needs it, it may overwrite it while translating source-level pseudoinstructions into real instructions.

So `$at` belongs more to the assembler's convenience machinery than to ordinary long-lived program state.

## Return-Value Registers: `$v0-$v1`

Registers `2-3` are:

```text
$v0-$v1
```

These are used for:
- function results,
- expression evaluation results,
- values returned from procedures.

So if a function computes a result, the conventional place to return it is usually `$v0`, and sometimes `$v1` as well if more space is needed.

## Argument Registers: `$a0-$a3`

Registers `4-7` are:

```text
$a0-$a3
```

These are used to pass arguments into a function.

So a normal call sequence looks like:
1. caller places arguments into `$a0-$a3`
2. caller executes `jal`
3. callee reads its inputs from those registers
4. callee returns its result in `$v0` or `$v1`

This is the first layer of procedure-call discipline.

## Temporary Registers: `$t0-$t9`

The temporary register families are:

```text
$t0-$t7
$t8-$t9
```

These are used for short-lived intermediate values.

Most importantly:

```text
they are not preserved across a function call
```

That means if the caller needs a temporary value after making a call, the caller must save it somewhere first.

So temporaries are ideal for:
- local intermediate expressions,
- short arithmetic chains,
- values that do not need to survive calls.

## Saved Registers: `$s0-$s7`

The saved register family is:

```text
$s0-$s7
```

These are preserved across a function call.

That means if a function uses one of these registers, it is responsible for restoring its original value before returning.

So these are appropriate for:
- longer-lived local state,
- variables that must survive nested calls,
- values the caller expects to remain unchanged after calling another function.

This creates a clear caller/callee division of responsibility:
- caller assumes `$t` registers may be destroyed
- caller assumes `$s` registers will be restored

## Kernel Registers: `$k0-$k1`

Registers `26-27` are:

```text
$k0-$k1
```

These are reserved for the operating system kernel.

User-level code should generally not rely on them.

Their existence reminds you that the register file is shared across:
- user code,
- calling convention rules,
- and privileged system-level mechanisms.

## Pointer and Control Registers

Several named registers play special structural roles.

### `$gp`

Global pointer.

Used to help access global or static data efficiently.

### `$sp`

Stack pointer.

This is one of the most important registers in procedure execution.

It points to the current stack region used for:
- spilled temporaries,
- local variables,
- saved registers,
- procedure-call bookkeeping.

### `$fp`

Frame pointer.

Used to reference the current stack frame in a stable way.

This is useful when stack layout is more complicated than simple push/pop movement.

### `$ra`

Return address.

This is register `31`.

When `jal` is executed, the machine stores the return location in `$ra`.
Then the callee can later return with:

```text
jr $ra
```

So `$ra` is the bridge between:
- procedure call,
- control transfer,
- and correct return to the caller.

## Preserved Across a Call

The most important column in a register-convention table is often:

```text
Preserved across a call?
```

This determines who is responsible for keeping a value alive.

### Preserved

Examples:
- `$s0-$s7`
- `$gp`
- `$sp`
- `$fp`

These are expected to remain valid across a call.

If the callee wants to use them, the callee must save and restore them.

### Not preserved

Examples:
- `$t` registers
- `$a` registers
- `$v` registers
- `$at`

These may be overwritten by the callee or by assembler expansion rules.

So if the caller wants to keep such a value, the caller must save it first.

## Why Calling Conventions Exist

Without a calling convention, one function would not know:
- where to find its arguments,
- where to place its result,
- which registers are safe to overwrite,
- and which registers must be restored before return.

That would make separate compilation and modular programming unreliable.

So the calling convention is a systems contract between:
- compiler,
- assembler,
- caller,
- callee,
- runtime,
- and operating system assumptions.

This is why register naming is not just cosmetic.
It encodes procedure-discipline expectations.

## The Deeper Relation Between Pseudoinstructions and Register Convention

These two topics fit together because both are forms of structured cooperation.

Pseudoinstructions are a contract between:
- programmer
- and assembler

Calling conventions are a contract between:
- caller
- callee
- compiler
- and runtime

Neither one is "the raw hardware alone."
They are both layers of disciplined interpretation built on top of the hardware ISA.

So a realistic understanding of assembly language requires both:
- the true instruction set,
- and the conventions that make that instruction set usable in real programs.

## Why This Matters for ML Systems

This may look far from ML systems, but the lesson is durable:

- low-level systems work depends on explicit conventions,
- interfaces between components matter as much as the primitive operations,
- and small hardware vocabularies become practical only when software layers add disciplined structure.

That same pattern appears later in:
- ABI rules,
- CUDA calling conventions,
- kernel launch interfaces,
- tensor layout conventions,
- runtime scheduling contracts,
- and distributed protocol boundaries.

So this MIPS topic is another early example of a larger systems truth:

```text
real systems are usable only when a small primitive mechanism is paired with strong conventions about how it must be used.
```
